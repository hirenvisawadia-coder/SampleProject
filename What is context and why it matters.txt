What is context and why it matters
Context is all the information an AI model can "see" when generating a response — the instructions, conversation history, documents, data, and tool outputs provided in its input window. Models are stateless: they have no memory beyond this window, so the quality of output depends directly on the quality and relevance of context supplied. Too little context produces generic or wrong answers; irrelevant or bloated context adds cost, latency, and can actually degrade accuracy by burying the important information.

How I manage context when building an AI solution
1. Prioritize relevance over volume. Include only what's needed for the task — retrieve the right document sections (RAG) rather than dumping entire files.
2. Structure the input. Clear sections, delimiters, and instructions placed prominently help the model weight information correctly.
3. Summarize and compress. Replace long histories or documents with concise summaries, keeping only decisions and facts that matter downstream.
4. Persist state externally. Store conversation state, user preferences, or intermediate results outside the model (database, files) and inject only the relevant slice per request.
5. Chunk large tasks. Break big jobs into steps, each with its own focused context, then combine results.

Example — working with limited context
When asked to analyze a document far too large to fit in the context window, I don't attempt to load it all at once. Instead I:
1. Chunk the document into logical sections,
2. Process each chunk independently — extracting key facts or partial summaries,
3. Aggregate those outputs in a final pass that synthesizes an overall answer.

For structured data (e.g., a 100,000-row spreadsheet), I sidestep the token limit entirely: rather than reading rows into context, I write code to compute the answer and only bring the results into context. The principle in both cases is the same — the model doesn't need all the data, it needs the right distillation of it. This keeps responses accurate, fast, and within limits regardless of source size.

How I approach memory and state management
Since AI models are stateless — each request starts fresh — memory must be engineered around the model, not expected from it. My approach:

Separate memory types. Short-term (current conversation), long-term (user preferences, past decisions), and working state (task progress, intermediate results) have different storage and retrieval needs.
Store state externally. Keep the source of truth in a database, file, or key-value store — never rely on the model to "remember." The model receives a relevant slice of stored state with each request.
Summarize, don't accumulate. Instead of replaying entire histories, periodically compress past interactions into concise summaries (key facts, decisions, preferences) and inject those. This keeps context small and relevant.
Retrieve selectively. Use search or semantic retrieval to pull only memories relevant to the current request, rather than everything ever stored.
Design for updates. Memory must be writable — when the user corrects a fact or changes a preference, overwrite the stored value so stale information doesn't persist.

Example — remembering information across interactions
Consider a multi-step data analysis assistant: a user uploads a dataset, asks a series of questions across many turns, and expects each answer to build on prior work. The model can't hold the dataset or every past result in context. The solution:

Persist the dataset and computed results to files — each analysis step writes outputs (cleaned data, aggregates, chart configs) to disk.
Maintain a compact state summary — a running record like "user filtered to Q3, excluded region X, defined revenue as net of returns" — injected into each new request.
Reload only what's needed — when the user asks a follow-up ("now break that down by month"), the system retrieves the relevant saved result and the state summary, not the full history.

The result: turn ten correctly honors a definition the user gave in turn one, without the context window growing unboundedly. The key insight is that memory in AI applications is an architecture problem — external storage plus smart retrieval — not a model capability.

Example — a customer support AI assistant
A support assistant handles a customer whose issue spans multiple sessions over several days — for instance, an unresolved billing dispute. Without memory, the customer must re-explain everything each time, and the AI may contradict earlier commitments.
The solution:

Persist a case record externally — customer details, issue summary, troubleshooting steps already tried, and promises made ("refund escalated on Monday") stored in a database keyed to the customer/ticket ID.
Inject a compact case summary at the start of each new session — not the full transcripts, just essentials: "Customer disputed a duplicate charge; card verification already completed; escalation pending with finance team."
Update memory after each interaction — new steps taken and outcomes are appended to the case record, and resolved items are marked closed so the summary stays current and short.

The result: when the customer returns three days later, the assistant immediately knows the history — it doesn't re-ask for verification or suggest steps already tried, and it honors the escalation promised earlier. This builds trust and cuts resolution time.
The key insight: memory in AI applications is an architecture problem — external storage plus smart retrieval and summarization — not something the model provides on its own.

How I track what an AI application is doing
AI components are probabilistic, so observability must go beyond traditional error logs — a failure often isn't an exception, it's a plausible-looking wrong answer. My approach covers three layers:
1. Logging

Log every AI interaction end-to-end: the full prompt (including retrieved context), model parameters, raw response, token usage, latency, and any tool calls made — all tagged with a request/trace ID.
Log the pipeline around the model: what was retrieved, what was filtered, what validation passed or failed. Most "AI bugs" turn out to be context bugs — the model was given the wrong input.
Structure logs (JSON with consistent fields) so they're queryable, and redact sensitive data before storage.

2. Monitoring

Operational metrics: latency, error/timeout rates, token cost per request, throughput — with alerts on thresholds.
Quality metrics: validation failure rate (e.g., malformed JSON output), retry rate, fallback usage, user feedback signals (thumbs down, corrections, escalations to a human).
Drift detection: track output distributions over time — if a classifier suddenly labels 40% of tickets "urgent" instead of the usual 10%, something changed (input data, model version, or prompt) even though nothing "errored."

3. Debugging

Reproduce with the trace: because I log the exact prompt and parameters, I can replay any failing request and inspect what the model actually saw.
Isolate the layer: check retrieval first (was the right context provided?), then the prompt (were instructions clear?), then the model output — in that order, since upstream issues are the most common cause.
Maintain a regression set: failing cases become test cases; every prompt or model change is evaluated against this suite before deployment.

How I know something went wrong
Three signal types, in increasing subtlety: hard failures (exceptions, timeouts, schema-invalid outputs — caught by automated validation on every response), soft failures (rising retry rates, fallback usage, negative user feedback), and silent failures (quality drift caught only by output-distribution monitoring and periodic sampling of responses against evaluation criteria — human or automated review of a random sample).
The core principle: with AI, you can't assume "no errors" means "working correctly." You must actively measure output quality, not just system health.

A hallucination example
While building a document Q&A assistant over company policy documents, the system was asked about a reimbursement limit. It confidently answered with a specific dollar figure and even cited a policy section — but the figure didn't exist anywhere in the documents. The retrieval step had returned only loosely related passages, and the model, rather than saying "I don't know," filled the gap with a plausible-sounding number. Classic hallucination: fluent, specific, and wrong.
How I identified the problem

User feedback flagged it — someone noticed the cited section didn't contain that number. This exposed that we had no automatic check.
Trace replay: because every request logged the exact prompt, retrieved chunks, and raw response, I replayed the failing request and saw the model's answer wasn't grounded in any retrieved text — the retrieval had missed the relevant document entirely.
Root cause split in two: (1) a retrieval gap — the policy used different terminology than the user's query, so the right chunk was never fetched; (2) a prompting gap — nothing forced the model to abstain when the context didn't contain the answer.

Steps to prevent recurrence

Grounding instructions: rewrote the prompt to require answers only from provided context, and to explicitly respond "not found in the documents" when the context is insufficient.
Citation verification: added a post-processing check that any quoted figure or cited section must actually appear in the retrieved chunks — mismatches are blocked and routed to a fallback response.
Improved retrieval: added query expansion (synonyms/rephrasing) and tuned chunking so terminology mismatches were less likely to cause empty-handed retrieval.
Confidence handling: low retrieval-relevance scores now trigger an "I couldn't find this" response instead of letting the model guess.
Regression testing: the failing question — and others like it — became part of an evaluation suite run before any prompt or model change ships.

The lesson
Hallucinations are rarely just a "model problem" — they're usually a system problem: weak retrieval plus permission to guess. The fix is to constrain the model to its evidence, verify outputs against that evidence automatically, and treat every discovered failure as a permanent test case.

How I optimize token and API-call usage
The guiding principle: send the model only what it needs, ask it only when needed, and never ask twice for the same thing.
1. Reduce tokens per request

Trim the context: retrieve only the most relevant chunks (better ranking, smaller top-k) instead of dumping whole documents; remove boilerplate, duplicate instructions, and stale conversation history.
Summarize history: replace long conversation logs with rolling summaries — key facts and decisions only.
Tighten prompts: concise instructions with a few well-chosen examples outperform bloated prompts; every example carries a permanent token cost, so keep only the ones that measurably help.
Constrain output: request structured, minimal responses ("return JSON with these 3 fields," "answer in under 100 words") — output tokens are typically the most expensive.

2. Reduce the number of API calls

Cache aggressively: identical or near-identical queries (FAQs, repeated classifications) return cached results instead of new calls; prompt caching reuses the static system-prompt portion across requests.
Batch: classify 20 items in one call with structured output rather than 20 separate calls.
Don't use AI for deterministic work: regex, lookups, and arithmetic handle many sub-tasks; the model is reserved for genuine judgment. Offloading a computation to code often eliminates entire calls.
Route by complexity: send simple, high-volume tasks to a smaller/cheaper model and reserve the large model for hard cases — a cascade where the small model escalates only when uncertain.

3. Avoid waste from failures

Validate before retrying: schema checks catch malformed output immediately; retries use a corrective prompt rather than repeating the full context.
Fail fast on bad input: validate upstream (empty documents, unsupported formats) before spending tokens on a doomed call.

Techniques that delivered the most impact
In practice, the biggest wins in order: (1) model routing — most requests don't need the most expensive model; (2) retrieval tuning — cutting retrieved context from ten chunks to the three genuinely relevant ones reduced input tokens dramatically and improved accuracy by removing noise; (3) caching — repeated queries are common in real usage and cost nothing to serve twice; (4) output constraints — structured short outputs over free-form prose.
The key insight: optimization and quality aren't opposed. Less irrelevant context and tighter outputs usually make answers better — the model focuses on what matters.

How I handle retrieval of chunks
1. Retrieval pipeline

Chunking strategy first: split documents along logical boundaries (sections, paragraphs) rather than fixed character counts, with modest overlap so ideas aren't cut mid-thought. Attach metadata (source, section title, date) to every chunk.
Hybrid search: combine semantic (embedding) search with keyword search — embeddings catch paraphrases ("reimbursement" vs. "expense claim"), keywords catch exact terms (product codes, names) that embeddings miss.
Over-retrieve, then filter: fetch a generous candidate set (e.g., top 20–30) knowing the first-pass ranking is imperfect, then narrow it in the reranking stage.

2. Reranking

Second-pass scoring: pass each candidate chunk with the query through a reranker (cross-encoder or LLM-based scoring) that judges actual relevance far better than raw embedding similarity — first-pass retrieval optimizes for speed, reranking optimizes for precision.
Apply business logic: boost recent documents over stale ones, prefer authoritative sources over informal notes, and deduplicate near-identical chunks.
Keep only the top few: typically 3–5 chunks make the final context. A relevance-score threshold matters too — if nothing scores well, it's better to trigger an "not found" response than to feed the model weak context it might hallucinate around.

3. Stitching into a coherent response

Order and label chunks: arrange them logically (by document order or relevance), clearly delimited with source labels so the model can attribute claims.
Ground the generation: instruct the model to answer only from the provided chunks, cite which source supports each claim, and state explicitly when the chunks don't contain the answer.
Resolve conflicts: when chunks disagree (e.g., old vs. new policy), instruct the model to prefer the most recent/authoritative source and note the discrepancy rather than silently picking one.

Trade-offs I consider

Recall vs. precision: retrieving more chunks reduces the risk of missing the answer but adds noise, cost, and can bury the relevant passage. Reranking is the lever that lets you have both — cast wide, keep narrow.
Chunk size: large chunks preserve context but dilute relevance scoring and waste tokens; small chunks rank precisely but may lack surrounding context. Section-based chunking with overlap is the usual middle ground.
Latency and cost vs. quality: reranking adds a step and milliseconds per query; for high-stakes Q&A it's worth it, for simple lookups it may not be.
Faithfulness vs. fluency: strict grounding produces safer but sometimes stilted answers; looser synthesis reads better but risks hallucination. For factual domains I bias firmly toward faithfulness.

The core principle: retrieval quality caps answer quality — no prompt engineering can rescue a response built on the wrong chunks, so most of the effort belongs before the model ever generates a word.

Comparing: Simple API calls vs. LangChain vs. LangGraph
1. Simple API calls (direct SDK)
You call the model provider's API directly and write your own prompt construction, retries, and control flow in plain code.

Strengths: complete control and transparency — you see exactly what's sent to the model; trivial to debug; no dependency churn; immediate access to new API features (tool use, prompt caching, structured outputs).
Weaknesses: everything is hand-built — retrieval, memory, multi-step orchestration — so complex workflows mean more code to write and maintain.
Best for: simple pipelines (one or a few model calls plus business logic) and production systems where reliability, cost control, and debuggability are the priority.

2. LangChain
A general orchestration framework offering pre-built components — chains, memory, retrievers — plus a large ecosystem of integrations (vector stores, model providers, tools).

Strengths: very fast prototyping — a working RAG pipeline in a few dozen lines; provider-agnostic, so swapping models is easy; the integration catalog saves significant glue code.
Weaknesses: layers of abstraction obscure what prompt was actually assembled, making debugging harder; historically frequent breaking changes; for simple tasks it adds complexity rather than removing it. Its linear "chain" model also strains when workflows need loops or branching.
Best for: proofs-of-concept and prototypes, especially when you need many integrations quickly.

3. LangGraph
A framework (from the LangChain team) for building stateful, graph-based agent workflows — you define nodes (steps) and edges (transitions), including cycles, branches, and human-in-the-loop pauses.

Strengths: the right abstraction for genuinely complex agent behavior — loops (retry until valid), conditional routing (escalate if uncertain), parallel branches, and persistent state/checkpointing so long-running workflows can pause and resume. Explicit graph structure makes complex flows easier to reason about than nested chains.
Weaknesses: steeper learning curve; overkill for linear pipelines; still inherits some ecosystem abstraction overhead.
Best for: multi-step agents that need cycles, branching, durable state, or human approval steps — e.g., a support agent that gathers info, tries tools, loops on failure, and escalates conditionally.

How I choose
The decision is essentially about control-flow complexity: if the workflow is a straight line with a couple of model calls — direct API calls, no framework needed. If I'm prototyping quickly and want batteries-included RAG or integrations — LangChain. If the workflow is a genuine state machine — loops, conditional paths, resumable state — LangGraph earns its complexity.
The pattern I follow: start with direct API calls by default, and only adopt a framework when the problem demonstrably demands it. Frameworks accelerate the first 80% but can obstruct the last 20% — many production systems end up as direct SDK calls with framework-inspired structure, keeping only the abstractions that pay for themselves.

AI across the software development process — three phases
1. Coding
The most mature use case. Specific examples:

Boilerplate and scaffolding: generating CRUD endpoints, data models, API clients, and config files from a short description — turning an hour of typing into minutes of reviewing.
Working in unfamiliar territory: when using a new library or language, asking AI for idiomatic examples ("how do I do connection pooling with this ORM?") is faster than piecing together docs and Stack Overflow answers.
Refactoring and explaining legacy code: pasting a convoluted function and asking for an explanation, then a cleaner rewrite — AI is particularly good at untangling code nobody remembers writing.

The discipline that makes it work: treat AI output as a draft from a fast junior developer — always review, never merge blind. It's most reliable on well-scoped, self-contained tasks and weakest when correctness depends on codebase-wide context it can't see.
2. Testing
Often the highest-leverage phase, because test writing is the work developers most readily skip:

Generating test cases from code: given a function, AI produces unit tests covering the happy path plus edge cases humans forget — empty inputs, nulls, boundary values, unicode, negative numbers. It's systematically thorough where humans are selectively thorough.
Test data generation: creating realistic fixtures — customer records with plausible variety (edge-case names, addresses, dates) instead of "test1, test2, test3."
Regression tests from bugs: pasting a bug report and the offending code, and asking for a test that reproduces the failure — locking the fix in place permanently.

Caveat: AI-generated tests can be superficially green — asserting what the code does rather than what it should do. The developer still has to verify the assertions reflect the actual requirements.
3. Requirements & design
Less automated, but valuable as a thinking partner:

Finding gaps in requirements: feeding a draft spec to AI and asking "what's ambiguous, missing, or contradictory?" reliably surfaces unhandled cases — what happens on concurrent edits? what if the payment succeeds but the confirmation email fails? It plays devil's advocate without ego.
Design trade-off exploration: describing a problem ("event-driven vs. request-response for this integration?") and getting a structured comparison of trade-offs — not to make the decision, but to make sure no major consideration was overlooked.
Drafting artifacts: turning rough notes from a design discussion into a structured design doc or user stories with acceptance criteria, which humans then correct and refine.

The common thread
In every phase, the pattern is the same: AI removes the blank-page problem and handles breadth (edge cases, boilerplate, alternatives) while humans provide judgment, context, and final review. The productivity gain is real, but it comes from augmentation — the phases where teams get burned are the ones where AI output ships without human verification.

An agent workflow I've implemented: a document-analysis agent with tool use
The problem
Users needed to ask questions over uploaded business documents (spreadsheets, PDFs, reports) — where answers often required computation, not just reading. "What was the month-over-month growth in Q3?" can't be reliably answered by a model reading numbers; it needs actual calculation.
The structure: an orchestrator agent with tools
I structured it as a single agent in a loop with a small set of well-defined tools, rather than a multi-agent system:

Plan step: the model receives the user question plus a manifest of available files (names, types, sizes — not contents) and decides what it needs.
Tool loop: the agent calls tools iteratively —

read_file(path, range) — inspect document structure or specific sections,
run_code(script) — execute Python for parsing, math, and aggregation in a sandbox,
search_chunks(query) — semantic retrieval over indexed text for unstructured questions.


Observe → decide: after each tool result, the agent decides the next action or concludes. Crucially, computation results come from code execution, never from the model "doing math."
Synthesize: final answer generated only from gathered evidence, with sources cited.
Guardrails around the loop: a hard cap on iterations (e.g., 10), per-tool timeouts, and schema validation on every tool call — malformed calls get one corrective retry, then fail gracefully.

What made it work well

Code execution as the workhorse tool. Delegating anything numerical to Python eliminated the largest error class (arithmetic hallucination) in one design decision.
Few, sharply-defined tools. Three tools with crisp descriptions outperformed an earlier draft with eight overlapping ones — the agent chose correctly far more often when choices were fewer.
Manifest-first, content-later. Giving the agent file metadata up front and letting it read selectively kept context small and made it scale to large files.
Full trace logging. Every plan, tool call, and result logged with a trace ID — indispensable for debugging why the agent took a wrong path.

Challenges faced

Looping behavior: early versions occasionally re-ran the same failing tool call repeatedly. Fix: include the error message of the previous attempt in context and enforce the iteration cap.
Overconfident conclusions: the agent sometimes answered after insufficient exploration. Fix: a prompt-level requirement to state what evidence supports the answer, which pushed it to gather before concluding.
Latency vs. thoroughness: each loop iteration adds seconds; we tuned the system prompt to prefer batching (one script computing several things) over many small steps.

The takeaway
The pattern that worked was a simple loop, strong tools, strict guardrails — most of the reliability came not from sophisticated agent architecture, but from constraining the agent: fewer tools, capped iterations, validated calls, and delegating precision work to code.

An agent workflow I've implemented: an automated support-ticket triage and resolution agent
The problem
Incoming support tickets arrived in high volume with wildly varying content — billing questions, bug reports, feature requests, account issues. Human agents spent significant time just categorizing, gathering context, and answering questions that were already covered in documentation. The goal: an agent that triages every ticket, resolves the routine ones, and escalates the rest with context attached.
The structure: a staged pipeline with an agent loop in the middle

Deterministic pre-processing (no AI): parse the ticket, extract customer ID and metadata, pull account status from the database. Anything rule-based happens in plain code first — cheaper and perfectly reliable.
Classification step: a small, cheap model classifies the ticket (category, urgency, sentiment) with structured JSON output. Low-confidence classifications route straight to a human.
Agent resolution loop: for categories marked auto-resolvable, a stronger model runs a tool loop:

search_kb(query) — retrieve relevant help-center articles,
get_account_info(customer_id) — fetch order/subscription details (read-only),
check_known_issues() — query the current incident/outage list.
The agent gathers evidence, then drafts a response grounded in what it retrieved.


Validation gate: the draft passes automated checks — every factual claim must trace to a retrieved source, no commitments the agent isn't authorized to make (refunds, credits), correct customer name. Failures route to a human instead of retrying indefinitely.
Human-in-the-loop by tier: routine responses (password resets, "where is my invoice") send automatically; anything touching money or account changes is drafted but queued for one-click human approval. Escalated tickets arrive with the agent's summary, gathered context, and suggested next step — so even "failed" automation saves the human time.

What made it work well

Cheap-model routing: the small classifier handled the high-volume decision at a fraction of the cost, reserving the expensive model for actual resolution work.
Read-only tools + approval gates: the agent could look anything up but couldn't change anything without human sign-off — this made deployment safe enough to ship early and expand gradually.
Escalation as a first-class outcome: designing "hand off gracefully with context" as a success path, not a failure, meant the system added value on 100% of tickets even though it fully resolved only a portion.
Feedback loop: human edits to drafted responses were logged and periodically reviewed to improve prompts and KB gaps.

Challenges faced

Overconfident answers on stale docs: the agent sometimes answered from outdated KB articles. Fix: recency metadata in retrieval, boosting current articles and flagging conflicts.
Sentiment blindness: early versions sent cheerful templated answers to furious customers. Fix: the sentiment classification now routes angry/high-value tickets directly to humans regardless of category.
Scope creep in responses: the model would occasionally offer goodwill credits unprompted. Fix: an explicit "authorized actions" list in the prompt plus a validation rule blocking any monetary commitment.

The takeaway
The architecture that worked was deterministic where possible, AI where necessary, human where risky — a staged funnel where each stage handles what it's best at. The agent loop was only one component; the reliability came from the routing, validation gates, and graceful escalation wrapped around it.

An agent workflow I've implemented: a research assistant that answers questions using web search
The problem
Users asked questions the model couldn't reliably answer from memory — recent events, current prices, product details. Instead of guessing, the AI needed to look things up before responding.
The structure: a simple search-and-answer loop

Decide: the model first checks — can I answer this from knowledge, or do I need to search? Stable facts get answered directly; anything current or uncertain triggers a search.
Search: the agent calls a web_search(query) tool with a short, focused query.
Read: it reviews the results, and if a snippet looks promising but incomplete, calls fetch_page(url) to read the full source.
Repeat if needed: if the first search didn't answer the question, it refines the query and tries again — up to a maximum of 3–5 attempts.
Answer: the agent writes the final response using only what it found, citing sources — and says plainly if it couldn't find a reliable answer.

What made it work well

The "decide first" step: not every question triggered a search — simple questions got instant answers, keeping the system fast and cheap.
A cap on attempts: limiting the loop to a few searches prevented endless retrying when information simply wasn't available.
Grounding rule: the answer had to come from the search results, not memory — this eliminated most hallucinated "facts."

Challenges faced

Repetitive searches: early versions retried nearly identical queries. Fix: instruct the agent to change its query meaningfully on each retry.
Trusting bad sources: the agent initially treated all results equally. Fix: prompt guidance to prefer official/original sources over forums and aggregator sites.

The takeaway
Even the simplest agent pattern — decide → act → observe → repeat → answer — needs guardrails: a cap on loops, a grounding rule, and a graceful "I couldn't find it" path. Those three constraints did more for reliability than any clever prompting.

Human in the Loop vs. Human on the Loop
The difference comes down to when the human gets involved: before the action, or after.
Human in the Loop (HITL): the AI cannot complete the action on its own — a human must review and approve each output before it takes effect. The human is a required checkpoint inside the workflow. AI proposes, human disposes.
Human on the Loop (HOTL): the AI acts autonomously, and the human supervises from outside — monitoring dashboards, sampling outputs, and intervening only when something looks wrong. The human is an overseer with a kill switch, not a gate on every action.
The trade-off is straightforward: HITL gives maximum safety but limits throughput to human review speed; HOTL scales to full automation speed but accepts that some errors will happen before anyone catches them.
When I'd use Human in the Loop
AI-generated code changes merged via pull request. An AI assistant drafts a bug fix or refactor, but nothing reaches the main branch until a developer reviews and approves the PR. This is the right pattern because the cost of a bad merge is high (broken builds, subtle bugs shipped to production), the AI lacks full codebase context, and the volume is low enough that review doesn't become a bottleneck. The same logic applies to an AI drafting customer refunds or account changes — anything irreversible or touching money gets a human approval gate.
When I'd use Human on the Loop
AI-powered ticket triage and labeling. An agent automatically categorizes, prioritizes, and routes hundreds of incoming support tickets or bug reports per day. No human approves each label — instead, the team monitors metrics (distribution of categories, misrouting complaints, confidence scores) and audits a random sample weekly. This works because individual errors are cheap and recoverable (a misrouted ticket just gets rerouted), the volume makes per-item review impossible, and monitoring catches systematic drift. Similarly, AI generating draft test cases in CI can run autonomously — a bad test fails visibly and gets fixed.
The rule of thumb
Choose based on cost of error × reversibility: if a mistake is expensive or irreversible (deployments, payments, merged code, customer-facing commitments), put the human in the loop. If mistakes are cheap and correctable, and volume is high, put the human on the loop with good monitoring. Mature systems often start with HITL and graduate to HOTL as trust and track record accumulate.
